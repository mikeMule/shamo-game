1. OVERVIEW
-----------
The withdrawal system allows users to cash out their spin earnings to TeleBirr.
Balance is derived from spin_results (amount_etb) where w-status = 'active'.
When admin releases a withdrawal, all user's spins are marked deactive so balance
shows 0 and prevents multiple withdrawal requests.

2. BALANCE LOGIC
----------------
- Available balance = SUM(amount_etb) FROM spin_results
  WHERE user_id = X AND "w-status" = 'active'
- w-status values:
  - 'active'  = counts toward balance (user can withdraw)
  - 'deactive' = paid out via withdrawal (balance shows 0)
- New spins default to w-status = 'active' (migration 012)
- No withdrawal subtraction from balance — balance is purely from active spins

3. DATABASE SCHEMA
------------------

3.1 spin_results."w-status" (migration 012)
   - Column: "w-status" VARCHAR(20) NOT NULL DEFAULT 'active'
   - Index: idx_spin_results_user_w_status (user_id, "w-status")
   - Purpose: Track which spins have been paid out

3.2 withdrawals table
   - status: pending | processing | completed | failed
   - amount_requested, fee_etb, amount_paid
   - phone_number, full_name (TeleBirr details)
   - processed_by, processed_at

4. MIGRATIONS (in order)
------------------------

012_spin_results_w_status.sql
   - Adds "w-status" column to spin_results
   - Default 'active'

013_get_active_spin_total_rpc.sql
   - RPC: get_active_spin_total(p_user_id UUID)
   - Returns SUM(amount_etb) WHERE user_id AND w-status='active'

014_get_spin_total_simple.sql
   - RPC: get_spin_total_simple(p_user_id UUID)
   - Fallback: SUM(amount_etb) for user_id only (no w-status filter)

015_release_mark_spin_deactive.sql
   - Documentation: release marks spins deactive

016_mark_user_spins_deactive_rpc.sql
   - RPC: mark_user_spins_deactive(p_user_id UUID)
   - Updates spin_results SET "w-status"='deactive' WHERE user_id AND w-status='active'
   - Returns rows updated (SECURITY DEFINER, bypasses RLS)

017_withdrawal_completed_mark_spins_deactive.sql
   - Extends on_withdrawal_status_change trigger
   - When status → completed: UPDATE spin_results SET "w-status"='deactive'
   - Runs atomically in same transaction as withdrawal update

5. DATABASE TRIGGER
------------------
Trigger: trg_withdrawal_status ON withdrawals
Function: on_withdrawal_status_change()

- pending → processing: deduct users.balance, add transaction
- pending|processing → failed: refund users.balance, add transaction
- pending|processing → completed: UPDATE spin_results SET "w-status"='deactive'
  WHERE user_id = NEW.user_id AND "w-status" = 'active'

6. API ENDPOINTS
----------------

6.1 Player (mini-app)
   GET  /api/player/{uid}/balance
        Returns { available_balance } for withdraw modal
   GET  /api/player/{uid}/balance-summary
        Returns available_balance, total_withdrawn, active_spins
   GET  /api/player/{uid}/withdrawals
        All withdrawals (for pending check)
   GET  /api/player/withdraw/config
        Returns min_withdrawal_etb (50), withdrawal_fee_pct (5)
   POST /api/player/withdraw
        Body: user_id, amount_requested, phone_number, full_name
        - Checks balance from spin_results (w-status='active')
        - Blocks if pending/processing withdrawal exists
        - Inserts withdrawal with status=pending

6.2 Admin
   GET  /api/withdrawals
        List with filters (status, search, pagination)
   GET  /api/withdrawals/{wid}
        Single withdrawal details
   PUT  /api/withdrawals/{wid}
        Update notes
   POST /api/withdrawals/{wid}/approve
        pending → completed (Release to Talabirr)
        - Updates total_withdrawn on users
        - DB trigger marks spins deactive
   POST /api/withdrawals/{wid}/complete
        processing → completed
        - Same as approve
   POST /api/withdrawals/{wid}/deny
        pending|processing → failed
        - Refunds users.balance via trigger

7. FRONTEND (index.html)
------------------------

7.1 Balance display
   - getAvailableBalance(): uses balanceSummary.available_balance only
   - No fallback to S.spins (spins include deactive, would show wrong amount)

7.2 Withdraw button states
   - Pending: disabled, shows "⏳ Pending"
   - Balance 0: disabled, shows "0 ETB · Min 50"
   - Balance > 0: enabled, opens modal

7.3 openWithdrawPopup()
   - If balance <= 0: toast "Available balance is 0 ETB", no modal
   - If balance < 50: toast "Minimum withdrawal is 50 ETB", no modal
   - If pending: show pending modal instead
   - Otherwise: open modal with form

7.4 Withdrawal flow
   - User taps Withdrawal → openWithdrawPopup
   - Fetches /api/player/{uid}/balance
   - Form: full name, phone (must match Telegram)
   - Submit → POST /api/player/withdraw
   - Success: refresh, show pending state

8. ADMIN FLOW (admin/withdrawals.html)
--------------------------------------
- Admin sees list of withdrawals
- Pending: "Release" button → POST /api/withdrawals/{id}/approve
- Processing: "Complete" button → POST /api/withdrawals/{id}/complete
- After release: DB trigger marks user's spins deactive
- User balance becomes 0; user sees "0 ETB · Min 50", withdraw button disabled

9. KEY FILES
------------
- api.py: balance helpers, withdraw endpoints, approve/complete
- index.html: profile UI, balance display, withdraw modal, button states
- admin/withdrawals.html: admin release UI
- migrations/012_spin_results_w_status.sql
- migrations/017_withdrawal_completed_mark_spins_deactive.sql

10. ERROR HANDLING
------------------
- "You already have a withdrawal in progress": one pending/processing at a time
- "Insufficient balance": requested > available (active spins only)
- "Minimum withdrawal is 50 ETB": configurable via platform_config
